## 层级定位

消息过滤和消息压缩的定义与实现放在 `core/memory_rag/memory/`，`agent_engine` 只负责在编排节点调用。

边界：
- memory 层负责 how（过滤算法、压缩算法、存储治理）
- agent_engine 负责 when（调用时机、顺序、失败降级）
- apps 层只声明策略，不感知实现细节

---

## 当前代码对齐（已落地）

当前 `MemoryGateway` 的短期记忆写入链路为：
1. 规范化：`normalize_content`
2. 噪声过滤：`NoiseFilter.is_noise`
3. 近窗去重：`DuplicateRecentFilter.is_duplicate`
4. 写入 Redis + TTL
5. 窗口压缩：`ShortTermWindowCompressor.trim_start` 后 `ltrim`
6. 按阈值触发 short→long consolidate

读取链路为：
1. 读取 short-term 最近窗口
2. 拼接 `role: content`
3. 读取 long-term 并聚合

## 模块拆分

### `memory/filters.py`

已承载：
- `normalize_content`
- `NoiseFilter`
- `DuplicateRecentFilter`

`manager.py` 内原 `_is_noise`、`_normalize_content`、`_is_duplicate_recent` 已下沉到该模块。

### `memory/compressor.py`

已承载：
- `ShortTermWindowCompressor`
  - `trim_start(current_length, max_turns)`：计算 `ltrim` 起始索引
  - `compress_messages(messages, max_turns)`：窗口压缩的纯内存版
- `SimpleSummaryCompressor`：规则摘要压缩（保留 recent + 系统摘要）
- `LLMSummaryCompressor`：LLM 摘要压缩（失败自动降级到 `SimpleSummaryCompressor`）
- `build_compressor(strategy)`：按策略构建压缩器（`window/simple_summary/llm_summary`）
- `build_tokenizer(provider)`：按策略构建 tokenizer（`char/tiktoken`）

### `memory/provider_protocols.py`

平台级能力契约：
- `MessageFilter`
- `MessageCompressor`
- `TokenizerProvider`
- `LongTermExtractor`
- `CompressionRequest` / `CompressionResult`

## 与 `manager.py` 的协作方式

- `append_short_term` 使用 filters + compressor 完成写入前过滤与压缩
- 压缩触发支持双阈值：
  - 轮次阈值：`compression_threshold`（未配置时回退 `short_term_max_turns`）
  - token 阈值：`compression_token_threshold` + `tokenizer_provider`
- `build_memory_context` 保持稳定接口，供 `agent_engine` 编排节点调用
- `consolidate_short_to_long` 继续复用 `normalize_content` 与 `NoiseFilter`

# 直接可用的开源方案

坦白说：**没有一个开源 SDK 能直接插入你这个架构做"生产级"消息压缩/过滤**，原因是这类能力高度依赖对话状态管理的具体实现（你用的是 Redis + LangGraph，别人的 SDK 几乎不会直接对接）。

但有几个方向的库值得认真评估：

---


## 方向一：LangChain Memory（最接近，但有坑）

python


```python
from langchain.memory import ConversationSummaryBufferMemory
from langchain_openai import ChatOpenAI

memory = ConversationSummaryBufferMemory(
    llm=ChatOpenAI(model="gpt-4o-mini"),
    max_token_limit=1000,        # 超限触发 LLM 摘要压缩
    return_messages=True,
)
```

**原理**：维护一个 moving summary + 最近 N 条原始消息，超过 token 上限自动调 LLM 压缩老消息。

**适配你的架构的工作量**：

python


```python
# 需要自己写桥接层，将 LangChain Memory 的输出写回你的 Redis
class LangChainMemoryBridge:
    def __init__(self, config: MemoryConfig):
        self._mem = ConversationSummaryBufferMemory(
            llm=llm_gateway.get_chat([], task_type="simple"),
            max_token_limit=config.compression_threshold * 200,
        )

    async def compress_messages(self, messages: list[dict]) -> list[dict]:
        # 灌入消息
        for m in messages:
            if m["role"] == "user":
                self._mem.chat_memory.add_user_message(m["content"])
            elif m["role"] == "assistant":
                self._mem.chat_memory.add_ai_message(m["content"])
        # 取出压缩后结果
        return self._mem.load_memory_variables({})["history"]
```

**结论**：可以复用它的压缩逻辑，但要自己写桥接，不能直接 plug-in。

## 方向二：mem0（你的 pyproject.toml 已经引入了）

python


```python
from mem0 import Memory

m = Memory()
# 自动提取、去重、压缩对话中的关键信息
m.add(messages, user_id="user_123")
result = m.search("理赔材料", user_id="user_123")
```

**这是你项目里最值得深度使用的**。mem0 做的事情：

- 自动从对话中提取事实（用 LLM）
- 去重和合并相似记忆
- 向量化存储，支持语义检索


**关键问题**：mem0 管的是**长期记忆提取**，不是短期消息窗口的压缩。它不直接操作你的 Redis 消息队列。

python


```python
# 在 update_memory 节点里接入
async def update_memory(state):
    # 现有 Redis 短期记忆
    await memory_manager.append_short_term(...)
  
    # mem0 处理长期记忆提取与去重（已有依赖，直接用）
    if config.long_term_enabled:
        m.add(
            [{"role": m.role, "content": m.content}
             for m in state["messages"][-4:]],
            user_id=f"{tenant_id}:{conversation_id}"
        )
```

## 方向四：tiktoken（token 计数，压缩判断必备）

python


```python
# pip install tiktoken
import tiktoken

enc = tiktoken.encoding_for_model("gpt-4o")

def count_tokens(messages: list[dict]) -> int:
    total = 0
    for m in messages:
        total += len(enc.encode(str(m.get("content", ""))))
        total += 4  # role/name overhead
    return total + 2  # reply primer
```

**这个强烈建议引入**。你现在的压缩触发条件是"轮次数"（`short_term_max_turns`），但实际上应该用 token 数来触发，因为每轮消息长度差异极大（用户说"好的"和贴一段材料，token 差 100 倍）。

对 `MemoryConfig` 的建议改动：

python


```python
@dataclass
class MemoryConfig:
    # 现有轮次限制保留，新增 token 限制
    compression_token_threshold: int = 3000   # 超过此 token 数触发压缩
    compression_threshold: int = 0            # 原轮次触发可设为兜底
```

## 综合建议

| 库 | 引入价值 | 使用方式 |
| --- | --- | --- |
| **tiktoken** | 高，生产必备 | 直接替换轮次计数为 token 计数触发压缩 |
| **mem0**（已引入） | 高，长期记忆直接用 | 在 `update_memory` 节点调 `m.add()` |
| **LangChain Memory** | 中，借用压缩逻辑 | 写桥接层复用 `ConversationSummaryBufferMemory` |
| **LlamaIndex** | 低 | 架构参考，不建议直接引入 |

实际上你现在的项目结构已经很清晰，自己实现 `filters.py` + `compressor.py` 的代码量不超过 200 行，反而比引入一个需要大量适配的 SDK 更可控。**唯一值得直接引入的新依赖是 tiktoken**，其他的用已有的 mem0 + 自实现即可。
