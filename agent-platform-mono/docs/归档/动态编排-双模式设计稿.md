# Agent 动态编排双模式设计稿（Command + PlanExecute）

## 1. 背景与结论

当前平台已具备可控动态编排的基础能力：`ToolRouter`、`checkpointer`、统一工具调用入口、多租户上下文透传。  
在此基础上，建议框架层同时支持两种编排模式，并向上层提供统一选择能力：

- `command`：基于节点路由的动态编排，低风险、低时延、可控性强
- `plan_execute`：基于任务规划与重规划的多步编排，适合复杂任务

核心结论：

- 可以同时提供两种模式给上层选择
- 但必须采用“统一合同 + 双引擎实现 + 策略化选择”架构
- 默认 `command`，`plan_execute` 通过灰度逐步放量

---

## 2. 设计目标

- 对上稳定：apps 仍通过 `factory() -> Graph`、`/agent/run`、`/agent/stream` 接入
- 对内可扩展：新增编排模式不影响 apps 合同
- 可治理：统一的租户隔离、工具权限、配额、日志、降级策略
- 可观测：能按模式比较质量、成本、时延、回退率
- 可恢复：任一模式都支持 checkpoint 恢复与断点续跑

---

## 3. 总体架构

### 3.1 统一抽象层（对上）

在 `AgentMeta` 中新增编排模式字段与策略字段：

```python
class AgentMeta(BaseModel):
    agent_id: str
    name: str
    description: str
    factory: Callable[[], Any]
    orchestration_mode: Literal["command", "plan_execute"] = "command"
    routing_mode: Literal["rule", "llm", "hybrid"] = "hybrid"
    fallback_mode: Literal["command", "finalize"] = "command"
    max_steps: int = 12
    max_replans: int = 2
```

说明：

- `orchestration_mode` 决定主执行引擎
- `fallback_mode` 控制复杂模式失败后的回退策略
- `max_steps/max_replans` 防止无限循环和成本失控

### 3.2 双引擎层（对内）

- `CommandEngine`：
  - 以 Router 节点输出 `next_node`
  - 通过条件边或 `Command(goto=...)` 动态跳转
  - 强调确定性与稳定降级
- `PlanExecuteEngine`：
  - `planner -> executor -> replanner -> finalize`
  - 支持根据执行结果修订剩余计划
  - 对复杂多步任务更友好

### 3.3 统一治理层（共享）

两种模式共用以下能力：

- 工具调用统一经 `tool_gateway.invoke()`
- 模型调用统一经 `llm_gateway` 与 `routing.select_model`
- 检索统一经 `rag_gateway`
- 日志统一经 `shared/logging`，透传 `tenant_id/conversation_id/thread_id/trace_id`

---

## 4. 统一状态合同

建议定义基础状态 `OrchestratorState`，两种模式共享核心字段：

```python
class OrchestratorState(TypedDict):
    messages: list[Any]
    tenant_id: str
    conversation_id: str
    thread_id: str
    trace_id: str
    step_count: int
    error_count: int
    memory_context: str
    rag_context: str
    route_decision: dict
    selected_tools: list[str]
    plan: list[dict]
    past_steps: list[dict]
    replan_count: int
    metadata: dict
```

约束：

- `tenant_id` 必填并参与工具调用、检索过滤、日志埋点
- 动态决策结果必须写入状态，确保可回放与可审计
- `step_count` 与 `replan_count` 必须有硬阈值

---

## 5. 模式选择策略

建议分三层决策，避免“全量手工配置”或“全量 LLM 决策”。

### 5.1 静态优先级

- Agent 级默认模式由 `AgentMeta.orchestration_mode` 指定
- 未配置时默认 `command`

### 5.2 运行时升级条件（可选）

在 `command` 执行中，若触发以下条件，可升级为 `plan_execute` 子图：

- 识别到多目标复合意图
- 预计需连续调用多个工具且步骤相关
- 用户显式要求“分步骤处理并给进度”

### 5.3 强制降级条件

`plan_execute` 触发下列任一条件，立即降级到 `command` 或 `finalize`：

- `planner` 输出无法解析或校验失败
- `replan_count` 超阈值
- 关键工具连续失败
- 请求超预算（token/时延/调用次数）

---

## 6. Command 模式设计

拓扑建议：

`memory_read -> decide_next_step -> (retrieve | reason | tool_exec | domain_node | finalize) -> memory_write`

关键点：

- Router 输出：`next_node/reason/confidence/guard_flags`
- 工具决策采用“LLM 建议 + ToolRouter 复核”
- 失败统一走 `fallback` 边，保证确定性收敛

适用：

- 单轮问答、短流程业务处理、延迟敏感场景

---

## 7. PlanExecute 模式设计

拓扑建议：

`planner -> validate_plan -> executor -> evaluate_step -> (replanner | finalize)`

关键点：

- `validate_plan` 必须做确定性校验：步骤可执行性、工具权限、参数完整性
- `executor` 每步完成后写入 `past_steps` 与 checkpoint
- `replanner` 只允许有限次数，防止无限重规划

适用：

- 跨多个系统、目标明确但链路较长的复杂任务

---

## 8. 接口与工厂设计

建议在 `agent_engine` 增加编排工厂：

```python
def build_orchestrator(meta: AgentMeta):
    if meta.orchestration_mode == "plan_execute":
        return build_plan_execute_graph(meta)
    return build_command_graph(meta)
```

建议在运行层增加统一入口：

- `run_agent(meta, state, config)`：同步结果
- `stream_agent(meta, state, config)`：流式事件

要求：

- 对上接口无感知模式差异
- 对下通过 graph metadata 标记 `mode` 与 `version`

---

## 9. 可观测与评估

统一埋点字段：

- `mode`, `agent_id`, `tenant_id`, `conversation_id`, `thread_id`, `trace_id`
- `route_reason`, `route_confidence`, `tool_count`, `replan_count`
- `latency_ms`, `token_in`, `token_out`, `error_code`, `degraded`

核心指标：

- 质量：任务达成率、用户负反馈率
- 效率：P50/P95 时延、平均 hops、token 成本
- 稳定：失败率、降级率、恢复成功率

建议按 `mode` 做 A/B 对比后再扩大 `plan_execute` 流量。

---

## 10. 安全与边界约束

- apps 层不直接依赖第三方 SDK，统一通过 `ai_core/tool_service/memory_rag` 抽象
- 工具执行必须带 `tenant_id`，并走统一鉴权与限流
- Prompt 变量白名单治理，禁止敏感信息落日志
- 模式切换不得绕过 `checkpointer` 与观测埋点

---

## 11. 落地路线图

### Phase 1：统一合同

- 补齐 `AgentMeta` 新字段与默认值
- 抽象 `OrchestratorState` 与公共埋点结构

### Phase 2：Command 标准化

- 固化 `decide_next_step` 输出契约
- 接入 `ToolRouter` 复核与 fallback 闭环

### Phase 3：PlanExecute 插件化

- 引入 `planner/validate_plan/executor/replanner` 子图
- 加入 `max_replans`、任务级 checkpoint 与强制降级

### Phase 4：灰度与评估

- 按 `tenant/agent` 小流量启用
- 以达成率/回退率/成本为阈值决策放量

---

## 12. 决策建议

当前阶段建议：

- 框架层立即支持双模式抽象，但默认仅启用 `command`
- `plan_execute` 先作为可选能力给少量 Agent 试点
- 用统一观测指标跑出数据后，再决定是否扩大为主模式

这能同时满足“上层可选”和“平台可控”两个目标，避免过早把全链路复杂性暴露给业务域。

## 13. 实现清单（可直接执行）

### 13.1 里程碑与交付物

- M1（合同层）：完成 `AgentMeta` 与统一状态合同，不改变现有 apps 接入方式
- M2（Command 引擎）：完成 `decide_next_step`、动态节点跳转、失败降级闭环
- M3（PlanExecute 引擎）：完成 `planner/validate_plan/executor/replanner` 子图
- M4（运行时选择）：完成模式选择器、升级/降级策略与统一运行入口
- M5（观测治理）：完成双模式统一埋点、指标看板字段与灰度开关
- M6（测试验收）：完成单元/集成/E2E 最小用例与回归检查

### 13.2 代码改造清单（按目录）

- `core/agent_engine/agents/registry.py`
  - 扩展 `AgentMeta`：`orchestration_mode`、`fallback_mode`、`max_replans`、`routing_mode`
  - 保持向后兼容默认值，不破坏现有注册流程
- `core/agent_engine/workflows/base_agent.py`
  - 抽离 `build_command_graph(meta)`，沉淀 `decide_next_step` 契约
  - 统一 fallback 边与 `step_count` 上限处理
- `core/agent_engine/workflows/`（新增）
  - 新增 `plan_execute.py`，实现 `build_plan_execute_graph(meta)`
  - 子节点：`planner`、`validate_plan`、`executor`、`replanner`、`finalize`
- `core/agent_engine/`（新增或扩展）
  - 新增 `orchestrator_factory.py`：`build_orchestrator(meta)`
  - 新增 `mode_selector.py`：静态模式、运行时升级、强制降级逻辑
- `core/agent_engine/tools/router.py`
  - 保持工具选择能力，补充与双模式共享的接口约束
  - 明确输出结构，便于 `command` 与 `plan_execute` 复用
- `core/tool_service/registry.py`
  - 补强调用参数校验，确保 `tenant_id` 与鉴权头必填
  - 统一错误归一化，供两种模式一致处理
- `core/ai_core/llm/client.py` 与 `core/ai_core/routing/router.py`
  - 为 planner/replanner 定义独立 task_type 路由策略
  - 保持统一模型调用入口，不在上层直接实例化 SDK
- `shared/config/settings.py`
  - 增加配置项：`ORCH_DEFAULT_MODE`、`ORCH_MAX_STEPS`、`ORCH_MAX_REPLANS`
  - 增加灰度配置：按 `tenant_id/agent_id` 启用 `plan_execute`
- `main.py`
  - `/agent/run` 与 `/agent/stream` 接入 `build_orchestrator(meta)`，对上接口不变
  - 补充模式与版本元信息到响应事件/日志上下文

### 13.3 数据结构与契约清单

- 新增统一状态模型 `OrchestratorState`（可放 `core/agent_engine/workflows/state.py`）
- 固化 Router 决策结构：
  - `next_node`
  - `reason`
  - `confidence`
  - `selected_tools`
  - `guard_flags`
- 固化 Plan 步骤结构：
  - `step_id`
  - `goal`
  - `tool_name`
  - `tool_input`
  - `expected_output`
  - `validation_rule`

### 13.4 观测与运营清单

- 日志字段统一：`mode/agent_id/tenant_id/conversation_id/thread_id/trace_id`
- 节点级埋点统一：`node_name/duration/error_code/step_count/replan_count`
- 指标最小集：
  - `orchestrator_requests_total{mode}`
  - `orchestrator_degrade_total{from_mode,to_mode}`
  - `tool_call_success_rate{mode}`
  - `avg_hops_per_request{mode}`
  - `p95_latency_ms{mode}`

### 13.5 测试清单

- 单元测试：
  - `mode_selector` 静态选择/升级/降级分支覆盖
  - `decide_next_step` 在不同输入下输出稳定
  - `validate_plan` 对非法计划拒绝并给出错误原因
- 集成测试：
  - `command` 模式完整链路：`memory -> reason -> tool -> finalize`
  - `plan_execute` 模式完整链路：`planner -> executor -> replanner/finalize`
  - `plan_execute` 失败后自动降级到 `command`
- E2E 测试：
  - `/agent/run` 与 `/agent/stream` 在两种模式均可成功返回
  - 多租户隔离字段全链路透传有效

### 13.6 验收标准（DoD）

- 对上接口不变：apps 无需修改 `factory` 合同即可运行
- 默认模式为 `command`，`plan_execute` 可按租户或 agent 灰度开启
- 任一模式失败都能确定性降级并收敛，不出现无限循环
- 关键链路日志与指标完整，支持按 `mode` 横向比较
- 通过 `ruff check`、`mypy`、最小测试集后方可合入
